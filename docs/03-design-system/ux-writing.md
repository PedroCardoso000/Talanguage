# UX Writing — Talanguage

## Purpose

This document defines the writing style for Talanguage product interfaces.

UX copy must help users act, understand progress, and keep practicing. It must not sound generic, childish, or overly motivational.

## Voice

Talanguage should sound:

- clear;
- encouraging;
- adult;
- practical;
- direct;
- supportive;
- focused on progress.

## Tone

Tone should adapt to the moment:

- onboarding: welcoming and simple;
- practice: action-oriented;
- feedback: constructive;
- errors: calm and clear;
- progress: motivating but not exaggerated;
- empty states: helpful and directional.

## Writing Principles

### 1. Tell the user what to do next

Good:

- “Choose a topic to start practicing.”
- “Review 5 cards to complete today's goal.”

Bad:

- “Your learning journey awaits.”
- “Unlock your maximum potential now.”

### 2. Keep copy short

Prefer one or two short sentences.

### 3. Avoid fake certainty

If feedback is simulated or approximate, do not present it as precise linguistic evaluation.

### 4. Use learning language consistently

Use terms from the glossary and domain docs.

### 5. Avoid shame

The product should correct and guide, not punish.

## Button Text

Button labels must use verbs.

Examples:

- Start practice;
- Finish session;
- Review cards;
- Submit answer;
- Try again;
- Continue learning;
- View progress.

Avoid vague labels:

- Go;
- Click here;
- Submit, when the action needs more clarity;
- Next, when the next step is unclear.

## Empty States

Empty states must include:

- what is missing;
- why it matters;
- what to do next.

Example:

“Your practice history is empty. Start a speaking session to begin tracking your progress.”

## Error Messages

Error messages must be:

- specific;
- calm;
- actionable.

Good:

“Could not start the practice session. Check your connection and try again.”

Bad:

“Something went wrong.”

## Feedback Copy

Feedback must be constructive.

Good:

“Good attempt. Try using complete sentences in your next answer.”

Bad:

“You made several mistakes.”

## Progress Copy

Progress copy must reinforce consistency.

Good:

“You completed 3 practice activities this week.”

Bad:

“You are becoming fluent fast.”

## AI-related Copy

When a feature is simulated, assisted, or AI-powered, the UI must not overpromise.

Allowed:

- “AI-assisted feedback”;
- “Practice suggestion”;
- “Simulated conversation”.

Forbidden:

- “Perfect fluency score”;
- “Certified pronunciation analysis”;
- “Guaranteed fluency”.

## Allowed

- simple action-oriented language;
- constructive correction;
- clear practice instructions;
- concise progress summaries;
- honest AI-assisted wording.

## Forbidden

- childish tone;
- exaggerated promises;
- shame-based correction;
- fake precision;
- generic motivational filler;
- long explanations inside UI.

## Definition of Done

UX copy is ready when:

- it is clear;
- it is short;
- it tells the user what to do next;
- it matches the product tone;
- it avoids overpromising;
- it uses domain terminology consistently.
